# AI 初创公司增长榜单设计稿

日期：2026-03-26

## 1. 项目目标

构建一个单页网站，展示基于 TrustMRR 数据计算得到的 AI 初创公司增长榜单。页面整体布局严格参考用户提供的手绘草图，视觉方向为简洁、克制、轻玻璃感的 Claude + iOS 风格，但不增加超出草图结构之外的装饰层。

第一版只实现首页，不做后台管理、用户登录或 CMS。

## 2. 页面结构

首页由 3 个全屏 section 组成，每个 section 占满一屏高度。

### Section 1

- 左侧展示标题“AI初创公司增长榜单”和一句简短说明
- 顶部保留轻量的 `AI Startups` 标识
- 右侧展示一个小型榜单预览框
- 底部中间有一个“查看榜单”按钮
- 鼠标悬浮到预览中的公司项时，展示信息卡片

### Section 2

- 展示完整 Top 20 榜单
- 布局参考 `www.unicor.ne` 右侧榜单区域，但严格服从手绘草图中的位置关系
- 榜单只显示三列：
  - `rank`
  - `公司`
  - `7天增长率`
- 右下角放置播放按钮，用于展示最近 7 天榜单变化回放
- 鼠标悬浮到公司项时，展示与 Section 1 相同结构的信息卡片

### Section 3

- 不使用外层大框
- 左上是 `Q&A`
- 中间纵向列出 3 条问答内容
- 下方展示“联系我”
- 旁边展示微信公众号二维码

## 3. 视觉设计原则

- 严格按照手绘草图的版式布局，不额外加壳层或多余容器
- 整体保持极简，强调留白和线框
- 使用非常轻的玻璃感和浅米色背景，避免复杂渐变和高饱和色
- 所有内容框使用细边框、圆角和轻微模糊，不做厚重阴影
- 文字层级清晰，但不过度装饰
- 移动端允许改为纵向堆叠，但保留相同的信息顺序和基本结构

## 4. 榜单数据定义

### 主指标

榜单主指标为真实的 `7天滚动收入增长率`。

公式如下：

```text
dailyRevenue[d] = totalRevenue[d] - totalRevenue[d-1]

recent7[d] = dailyRevenue[d] + dailyRevenue[d-1] + ... + dailyRevenue[d-6]

previous7[d] = dailyRevenue[d-7] + dailyRevenue[d-8] + ... + dailyRevenue[d-13]

growth7d[d] = (recent7[d] / previous7[d] - 1) * 100
```

其中 `totalRevenue` 来源于 TrustMRR 的 `revenue.total`。

### 入榜门槛

```text
previous7 >= 1000 美元
```

即只有前 7 天收入总和不低于 1000 美元的项目才参与排序。

### 平局处理

先看 `28天增长率`，再看 `最近7天收入绝对值`。

28 天增长率公式：

```text
growth28d = (recent28 / previous28 - 1) * 100
```

如果历史数据不足，无法计算 `growth28d`，则暂时退化为只比较 `recent7`。

### 历史不足处理

- 第 1 天到第 14 天：无法算出真实 `growth7d`
- 第 15 天起：可以首次计算真实 `growth7d`
- 第 57 天起：可以完整计算 `growth28d`

在历史不足期间，页面展示“计算中”。

## 5. 卡片字段规则

悬浮卡片字段采用“固定字段 + 条件字段”模式。

### 固定显示

- `产品描述`
- `目标用户`

### 条件显示

- `是否可售`
- `报价`
- `技术栈`

规则：

- 对应字段存在时才显示
- 字段不存在时直接隐藏该行
- 不做推断，不填充伪数据

## 6. 数据来源

全部数据来自 TrustMRR。

### 榜单与收入快照

使用 TrustMRR API 获取：

- `slug`
- `name`
- `icon`
- `rank`
- `growth30d`
- `revenue.total`
- `revenue.last30Days`

### 悬浮卡片详情

优先通过详情接口获取：

- `description`
- `targetAudience`
- `onSale`
- `askingPrice`
- `techStack`

如果某字段不存在，则前端不显示该字段。

## 7. 技术方案

### 前端框架

采用 `Next.js` 构建整站。

原因：

- 当前项目是空目录，适合从零建立单体前后端结构
- 页面、聚合接口、数据读取逻辑可以统一放在一个工程中
- 便于后续扩展静态生成、服务端渲染、定时任务入口和部署

### 数据层

定义 3 类数据：

#### 1. `startups_daily_snapshots`

按天保存列表快照，至少包含：

- `date`
- `slug`
- `name`
- `rank`
- `revenue.total`
- `revenue.last30Days`
- `growth30d`

#### 2. `startup_details_cache`

缓存 startup 详情，至少包含：

- `slug`
- `description`
- `targetAudience`
- `onSale`
- `askingPrice`
- `techStack`
- `updatedAt`

#### 3. `leaderboard_result`

由服务端根据历史快照聚合得到，包含：

- 排行名次
- 公司信息
- `growth7d`
- `recent7`
- `growth28d`
- 历史是否足够

前端页面只读取聚合结果，不在浏览器里计算历史收入。

## 8. 交互设计

### Section 1 榜单预览

- 只作为榜单预览入口，不展示完整 20 条
- hover 公司项时弹出卡片
- 点击“查看榜单”滚动到 Section 2

### Section 2 Top 20 榜单

- 默认只展示 `rank / 公司 / 7天增长率`
- hover 公司项时弹出卡片
- 卡片内容与 Section 1 共用同一组件

### 最近 7 天变化回放

播放按钮位于 Section 2 右下角。

点击后按天展示最近 7 天榜单变化：

- 每一帧表示某一天的榜单状态
- 展示名次变化和顺序变化
- 如果某天某个项目仍无足够历史，则该项显示“计算中”

第一版只做基础时间回放，不做复杂控制器。

## 9. 限流策略

TrustMRR API 限流为每个 API key `20 requests/minute`。

因此采用以下策略：

- 每天固定时点抓取一次列表快照
- 详情接口按需请求，并缓存结果
- 页面请求时优先读本地聚合结果和详情缓存
- 避免在用户 hover 时直接无缓存地批量请求外部 API

这样可以显著降低请求频率，避免触发限流。

## 10. 部署方案

### 网站部署

网站部署在中国大陆腾讯云服务器。

### 出海访问

由于 TrustMRR 为海外站点，国内服务器可能出现访问不稳定，因此服务器需要具备独立的出海能力，例如：

- `OpenVPN`
- `sing-box`
- `Clash`

只让 TrustMRR 相关流量走代理，其余站点流量保持直连。

### 密钥管理

TrustMRR API key 只保存在服务端环境变量中，不暴露到浏览器。

## 11. 错误处理

- API 请求失败时，页面继续显示上次成功聚合的缓存数据
- 若无缓存可用，则榜单区域显示加载失败提示
- 某公司详情拉取失败时，不影响榜单主体展示
- 某字段缺失时，卡片仅隐藏对应字段，不显示错误文案

## 12. 测试策略

第一版至少覆盖以下验证：

- 7 天滚动收入增长率计算正确
- 入榜门槛判断正确
- 平局排序规则正确
- 历史不足时返回“计算中”
- 详情字段缺失时卡片正确隐藏对应行
- 排行榜回放能按天输出最近 7 天结果
